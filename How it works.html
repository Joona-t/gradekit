Overview

What gradekit is

A tiny command-line tool that does one thing well: diagnose a shot's color from a single frame and produce a corrective grade as a portable .cube 3D LUT.

You point it at a video or image. It grabs a frame, measures the white balance, exposure, clipping and (optionally) skin tone, prints a readable diagnosis, and writes a LUT that is the fix. No accounts, no cloud, no paid API — everything runs on your machine with numpy, Pillow and ffmpeg.

What goes in

  • A video (.mov, .mp4, …) or an image
  • Optionally: a region you know is neutral gray/white
  • Optionally: a skin sample region

What comes out

  • A .cube LUT (White Balance + Exposure + gentle Contrast)
  • A plain-language + Lumetri-term diagnosis printed to the terminal
  • Optionally: a before/after preview PNG
The output is editor-agnostic. A .cube is the industry LUT format — it loads in DaVinci Resolve, Final Cut Pro, After Effects, Photoshop, OBS, ffmpeg and Premiere alike. Only the wording of the report ("Lumetri Basic") leans Premiere; the fix itself goes anywhere (see Applying the LUT).

Quick start

Install & run

Requires Python 3.11+, with numpy and Pillow. For video inputs you also need ffmpeg/ffprobe on your PATH (images don't). OpenCV is optional — it only upgrades automatic face detection; without it, gradekit falls back to a pure-numpy skin detector, so a missing dependency never breaks it.

Get it

# clone
git clone https://github.com/Joona-t/gradekit.git
cd gradekit
python3 -m venv .venv && . .venv/bin/activate
pip install numpy pillow            # the only hard deps

Run it (three ways)

# 1) straight from the repo
python3 -m gradekit analyze shot.mp4

# 2) the bundled launcher, from anywhere
/path/to/gradekit/bin/gradekit analyze shot.mp4

# 3) a clean isolated install
pipx install .                      # add '.[face]' for OpenCV face detect
gradekit analyze shot.mp4

The one-liner that does the useful thing

python3 -m gradekit analyze take2.mov --neutral 100,100,60,60 --lut look.cube --preview ba.png

That marks a known-neutral patch for precise white balance, writes look.cube, and saves a side-by-side ba.png so you can eyeball the correction before touching your editor.

How it works

The color science, written out

Nothing here is a black box. numpy is used as a fast calculator only — every transform is implemented straight from its definition. Here's the whole pipeline.

linear-light operations display / encoded operations baked output

1 · Work in linear light, not in sRGB

An 8-bit image is gamma-encoded: the stored numbers are perceptually spaced, not proportional to light energy. White balance and exposure are physical operations on light — they multiply photon counts — and multiplication is only meaningful where numbers are proportional to light: linear space. So gradekit always does un-gamma → math → re-gamma. Scaling raw sRGB numbers directly is the single most common mistake, and the reason amateur grades look muddy.

2 · White balance = per-channel gains (von Kries)

A neutral surface should read R = G = B in linear light. From a patch you mark with --neutral, gradekit computes a gain per channel that neutralizes it, using the patch's luminance (Rec.709 weights 0.2126/0.7152/0.0722) as the target so brightness doesn't change. With no patch it falls back to gray-world — assume the whole scene averages to gray — at lower confidence.

3 · Gains → approximate Lumetri Temperature/Tint

Lumetri's Temp/Tint scale is proprietary, so this is an explicit, calibrated approximation: Temperature from the blue-vs-red correction needed (boosting blue ⇒ the source was warm ⇒ a cooler, negative Temperature), Tint from green-vs-magenta. Treat the numbers as a strong starting point, not exact Kelvin.

4 · Exposure & tone from the histogram

Clipping is detected per channel in display space — any channel pinned at the ceiling/floor has lost recoverable detail. Exposure direction is computed in linear as log2(target / median) toward an 18% mid-gray anchor.

5 · The LUT is the grade

gradekit samples a 33³ lattice of input colors and runs every node through the exact same pipeline. The order is the whole point:

encoded input
  → linearize                 # un-gamma; now we're in light
  → per-channel WB gains       # linear, multiplicative
  → exposure × 2^stops         # linear, multiplicative
  → re-encode to sRGB          # back to display
  → gentle contrast S-curve    # in ENCODED space, pivots on mid-gray
  → clamp to [0,1]

The before/after preview's "after" image is produced by applying the actual baked LUT with hand-written trilinear interpolation — the same thing Premiere does internally — so it honestly previews what you'll see in your editor. A ground-truth test injects a known cast + exposure into a synthetic frame and asserts gradekit recovers the inverse.

The [baked] vs [dial] split

The .cube already contains White Balance + Exposure + a gentle Contrast curve.

  • [baked] numbers (Temperature, Tint, Exposure) are shown so you can understand or hand-tweak the look — don't also dial them, or you'll double-apply.
  • [dial] numbers (Highlights, Whites, Shadows, Blacks) are extra region moves not in the LUT — add them by hand on top if you want them.
  • Skin is diagnostic only and never baked — hue/saturation choices for skin are yours.

Reference

CLI reference

There's a single command, analyze:

gradekit analyze <input>          # a video or an image
FlagDefaultWhat it does
--t SECONDS~10% inTimestamp of the frame to grab (video only).
--neutral x,y,w,h—A region you know should read neutral. Drives precise white balance. Pixels from the top-left; a gray card or white wall is ideal.
--skin x,y,w,hautoA skin sample region for the (diagnostic-only) skin check.
--lut PATH./<stem>.cubeWhere to write the LUT.
--size N33LUT cube size (min 2). 33 is the standard.
--preview PATH—Write a before/after PNG.
--contrast F0.10Strength of the gentle contrast S-curve baked in (0 = none).

Exit codes: 0 success, 1 the LUT failed its format validation, 2 a bad argument or a friendly error (missing file, no ffmpeg, …). Useful when scripting or letting an agent branch on the result.

Output

Reading the report

A typical run prints something like this:

gradekit — shot.mp4 (video frame @ 1.40s) ──────────────────────────────────────────────────── The .cube bakes: White Balance + Exposure + gentle Contrast. WHITE BALANCE [method: neutral-patch · confidence: high] ✗ Warm/amber cast. why: sample reads unbalanced in linear light (gains 0.80 / 1.04 / 1.49 for R/G/B). Lumetri → Basic [baked]: Temperature -62 Tint +3 EXPOSURE ✗ Underexposed — midtones sit below mid-gray. why: median luma is about +0.3 stops under the 18% target. Lumetri → Basic [baked]: Exposure +0.3 ✗ 4.2% highlights blown · 4.2% shadows crushed. Lumetri → Basic [dial]: Highlights -34 Whites -21 Shadows +34 Blacks +21 SKIN [sample: explicit --skin · judged post-WB] ✓ Skin hue 23° (target 20–40), saturation 43% — healthy. LUT ✓ wrote shot.cube (33×33×33, 35937 rows, format valid)

Read it top-down: the cast and its direction, the exposure move, what's clipping, whether skin landed in a healthy range, and confirmation the LUT validated. Everything tagged [baked] is already in the .cube; [dial] values are optional manual additions.

In your editor

Applying the LUT (any editor)

The .cube is a standard 3D LUT. Drop it in wherever your tool takes a LUT:

ToolWhere
Premiere ProLumetri Color → Creative → Look → browse to the .cube. (Or type the printed Basic values.)
DaVinci ResolveCopy into a LUT folder → right-click a node → LUT, or drag the .cube onto a node.
Final Cut ProAdd the Custom LUT effect → Add Custom LUT… → import the .cube.
After EffectsEffect → Utility → Apply Color LUT.
PhotoshopLayer → New Adjustment Layer → Color Lookup → Load 3D LUT.
OBSSource/scene Filters → Apply LUT.
ffmpeg (headless)ffmpeg -i in.mp4 -vf lut3d=look.cube out.mp4
 
# burn the grade into a file with no GUI at all
ffmpeg -i in.mp4 -vf lut3d=look.cube -c:a copy out.mp4
Tip: in a node/layer-based tool, apply the LUT on an adjustment layer / shared node spanning the whole timeline so one grade covers every clip from that camera/lighting setup.

Automation

Using it with your agents

gradekit is built to be driven by AI coding agents (Claude Code, Codex, and friends). It's a plain subprocess with structured stdout and a file artifact — exactly what an agent can call and reason about.

The contract

  • Invocation: a single subprocess, no interactivity, no prompts.
  • Reads: the input video/image (+ optional region coords).
  • Writes: a .cube file (and optional preview PNG) — never overwrites source media.
  • Returns: the human-readable diagnosis on stdout + an exit code (0/1/2) to branch on.
  • Never touches the network. No API key. Ever. Transcription/grading are 100% local — an agent should not be asked to supply credentials.

The exact call to give an agent

Point at the tool's own venv interpreter so there's no activation step. Pattern:

/path/to/gradekit/.venv/bin/python -m gradekit analyze <video|image> \
  --lut /tmp/look.cube \
  --preview /tmp/before_after.png \
  --neutral x,y,w,h        # optional; precise white balance \
  --t 5                     # optional; which frame (seconds)

A capsule you can paste into an agent's context

Drop this verbatim into a system/task prompt so the agent knows the tool exists and how to call it:

TOOL: gradekit (local color grading, no API)
When footage looks warm/cool/over/under-exposed, run:
  /path/to/gradekit/.venv/bin/python -m gradekit analyze <file> --lut out.cube --preview ba.png
It prints Temperature/Tint/Exposure (Premiere Lumetri terms) and writes a
.cube LUT. The .cube applies in ANY editor (Resolve/FCP/AE/OBS/ffmpeg), e.g.
  ffmpeg -i in.mp4 -vf lut3d=out.cube out_graded.mp4
Pass --neutral x,y,w,h if a gray/white patch is known (more accurate).
Never add an API key — gradekit is 100% local.

How an agent typically uses it

  1. Notices footage with a color problem (or is told "fix the color").
  2. Runs analyze with --preview, reads the printed diagnosis + checks the before/after PNG.
  3. Either burns the LUT in headlessly with ffmpeg -vf lut3d=…, or hands the .cube + Lumetri values to premiere-grade-bot to apply inside Premiere.
  4. Branches on the exit code (re-run with a different --t frame if needed).
Fleet note. gradekit is documented alongside its sibling cutkit (transcript-aware auto-cut + captions) in a single agent-facing handoff file — hand that to any agent and it knows the whole video toolchain. gradekit is the color stage; cutkit is the edit/caption stage.

Companion

premiere-grade-bot

gradekit tells you the fix and bakes the LUT; premiere-grade-bot applies it in Adobe Premiere Pro for you, hands-free, via screen automation. It's a Claude Code skill (uses the computer-use tools) that:

  • creates an adjustment layer spanning the whole sequence,
  • types gradekit's measured Temperature / Tint / Exposure / Contrast straight into Lumetri Basic (no slider dragging) — or loads the baked .cube via Creative → Look,
  • verifies every step with a screenshot, and aborts rather than clicking blind.

It lives in the repo at skills/premiere-grade-bot/SKILL.md; install by copying that folder into ~/.claude/skills/ and asking your Claude Code session to "apply the grade in Premiere." It's Premiere-specific — for every other editor, just drop the .cube in by hand (see Applying the LUT) or burn it with ffmpeg.

Honesty

Limits & assumptions

  • SDR / Rec.709-ish material is the assumption. Frames are analyzed at 8 bits; HDR / log / 10-bit nuance isn't modeled.
  • Video decode uses ffmpeg's color-matrix/range defaults. Great for relative diagnosis (warm? clipping?); absolute numbers on exotic footage may drift.
  • Temp/Tint numbers are approximate — a calibrated mapping, not exact Kelvin.
  • Gray-world (no --neutral) is a guess and can be fooled by strongly colored scenes. Mark a neutral patch when you can.
  • Auto skin detection without OpenCV uses a YCbCr heuristic — pass --skin for reliability.

It's one frame, one honest corrective grade — a strong, explainable starting point, not a colorist replacement.
